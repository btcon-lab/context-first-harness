# Changelog

## 0.6.0 (2026-07-24)

v0.5.1이 "연기" 결정을 PROGRESS에 남기게 했다면, 이번은 **그 연기 결정을 조건 충족 시 실제로 꺼내 쓰는** 장치다. 결정을 남겨도 다시 수면 위로 올려주는 게 없으면 결국 잊히고, 그러면 조용한 스킵과 다를 바 없어진다.

- **연기된 운영 결정 게이트** (`execution-modes.md`) — 오케스트레이터가 매 실행 Phase 0에서 PROGRESS의 "하네스 운영 결정"을 읽고, 연기된 항목의 실행 조건이 충족됐는지 확인한다(예: 제어 루프 "M3 착수 시" ↔ 지금 구현 단계 진입). 충족되면 실행을 이번 회차의 선행 작업으로 올린다. **조건 판정("지금이 M3인가")은 오케스트레이터가 단정하지 않고 사람에게 확인받는다** — 판단을 없애는 게 아니라 잊히지 않게 올려주는 것이 목적이다.
- **정본에서 센서 시드 도출** (`control-loop.md`) — 연기된 제어 루프를 설치할 때 센서 설정을 백지에서 쓰지 않는다. `seam`(정본 파일명)·`rules`(Rule ID 목록)·`measures`(검증 항목 assertion)는 정본에서 자동 도출하고, `match`는 설계 경로로 반자동, **`command`만 사람이 채운다.**
- **`command` 자동 생성 금지** — 존재하지 않는 명령을 추측해 넣으면 훅이 그것을 실행해 `SENSOR_ERROR`를 양산하고, 그것을 통과로 처리하면 "미측정을 정상으로 읽지 않는다"(v0.3.0)가 깨진다. **자동 설치는 "판단 없는 실행"이며, v0.5.1이 없앤 조용한 스킵의 거울상이다** — 안 만드는 것도, 아무렇게나 만드는 것도 둘 다 판단이 빠진 것이다. `command`가 빈 센서는 설정에 남되 훅이 건너뛴다(실행할 게 없으므로). 빈 `command`가 실제로 건너뛰어지고 `_todo` 필드가 파싱을 깨지 않는 것을 픽스처로 확인했다.
- `seam-sensors.template.json`에 `command:""` + `_todo` 미확정 엔트리 예시 추가.

이 게이트는 문서만으로는 검증되지 않는다 — 실제로 이 하네스가 구현 단계(M3)에 진입해 게이트가 작동하고 센서가 시드되는지는 그때 확인한다.

## 0.5.1 (2026-07-24)

실제 프로젝트에 하네스를 구성해보니(9 seam) 제어 루프가 아예 검토되지 않고 빠졌다. `seam-sensors.json`도 훅도 없었고, 안 만든 사유도 어디에도 없었다. 원인은 SKILL.md가 제어 루프를 `(옵션)`으로만 표시한 것이다 — "옵션"이라고만 하면 하네스는 판단 없이 조용히 스킵한다.

- **제어 루프를 "조용한 스킵"에서 "명시적 결정"으로** — 설치 / 연기 / 생략 중 하나를 정하고 그 판단을 PROGRESS에 남기도록 바꿨다. 하네스 구성만 끝난 시점(구현 착수 전)은 대개 **연기**이며, 이때 "그래서 안 만든다"가 아니라 "구현 단계에서 설치"를 다음 할 일로 등재하는 것이 핵심이다 — 그래야 구현 단계에서 잊히지 않는다.
- `control-loop.md`에 "쓸지 말지 — 결정을 남긴다" 판단 기준표 추가(설치/연기/생략 조건).
- `PROGRESS.template.md`에 "하네스 운영 결정" 섹션 추가 — 제어 루프·Stop 훅 결정을 적을 자리. 자리가 없으면 하네스가 어디 적을지 몰라 스킵한다.
- `verification.md`: 제어 루프 결정이 PROGRESS에 기록됐는지를 구조 검증 항목에 추가. 아무 언급이 없으면 검증 실패로 본다(= 아무도 검토하지 않았다는 뜻).
- SKILL.md 산출물 체크리스트: "(옵션) 설치"에서 "설치·연기·생략 중 하나로 결정하고 사유 기록"으로.

## 0.5.0 (2026-07-23)

제어 루프를 추가했다. 계약을 세우고 팀을 만들어도 **작업이 진행되는 동안 계약이 지켜지는지 아무도 안 보면** 이탈은 통합 시점에야 드러난다. 그때는 이미 오염이 퍼진 뒤다.

- **4층 제어 루프** (`references/control-loop.md`) — `실행 → 관측 → 판정 → 교정`. 각 층은 하나의 책임만 지고, 상태를 바꾸는 층은 실행뿐이다. 층끼리 직접 호출하지 않고 훅이 사건과 결과를 함께 넘긴다 — 한 층이 실패해도 나머지가 멈추지 않는다.
- **센서 훅** (`scripts/seam-sensor-run.sh`) — PostToolUse 훅. 파일이 바뀌면 관련 센서만 골라 실행하고 관측 리포트를 남긴다. **훅은 측정만 한다.** 종료코드 1이 곧 위반은 아니므로(테스트 환경 문제일 수 있다), 정본과 대조하는 판정은 Claude가 한다. 0.3.0의 "측정 ≠ 판정" 원칙을 코드에서도 지킨다.
- **센서 설정** (`assets/seam-sensors.template.json`) — 센서마다 `seam`과 `rules`를 명시해 판정 시 대조 기준이 바로 나오게 했다. `match` 패턴으로 변경 범위에 걸린 센서만 실행한다(매번 전부 돌리면 느려서 아무도 안 쓴다).
- **타임아웃 폴백** — `timeout`/`gtimeout`이 없는 환경(기본 macOS)에서 제한이 조용히 무력화되던 문제를 순수 bash 폴백으로 해결했다. 제한이 안 걸리면 센서가 세션을 붙잡는다. 초과 시 `SENSOR_ERROR`로 기록한다.
- **에스컬레이션 규칙** — 센서 3회 연속 `SENSOR_ERROR`(검사 체계 자체의 문제), 미분류 관측(계약에 규칙 없음 → G 후보), 같은 Rule ID 반복 위반(완화 여부는 사람 판단).
- 출력은 stdout이 아니라 **파일**로 남긴다 — 훅 출력 처리 방식이 버전마다 달라도 결과가 보존된다.
- 제어 루프는 **선택 기능**이다. `seam-sensors.json`이 없으면 훅은 아무것도 하지 않는다. seam이 적은 프로젝트에는 과하다.

검증: 설정 없음 / 매칭 없음 / 정상 실행 / 명령 없음 / 타임아웃 5가지 경우를 픽스처로 확인했다.

## 0.4.0 (2026-07-23)

정본을 누가 언제 바꿀 수 있는지를 규정했다. 판정 체계(0.3.0)가 "무엇이 문제인가"를 정했다면, 이번은 "그래서 누가 고치는가"다.

- **두 루프 분리** — 검증에서 문제가 나오면 할 일이 두 갈래로 갈린다. V(구현 위반)는 **구현**을 고치는 즉시 교정 루프, G·S·D는 **정본**을 고치는 계약 개선 루프다. 한 흐름으로 처리하면 계약이 조용히 침식된다.
- **정본 변경 승인 게이트** — 계약은 아키텍처 결정이다. 에이전트가 검증 결과만 보고 정본을 고치면 당장은 QA가 통과하지만 **"구현에 맞춰 계약을 낮추는" 방향으로 흘러가고**, 정본은 코드의 사후 요약으로 전락한다. QA는 제안만 하고, 소유 에이전트가 초안을 쓰고, **사람이 승인해야** 반영된다. 승인 없이 가능한 것은 `[TODO]` 최초 기입·오타·서식·링크 교정뿐이다.
- **규칙 완화를 가장 엄격히** — 반복 위반은 규칙이 틀렸다는 뜻일 수도, 팀이 못 지킨다는 뜻일 수도 있다. 전자면 완화가 맞고 후자면 완화는 문제를 덮는 것이다. 정반대 처방이므로 사람이 판단한다.
- **재시도 정책** — 같은 입력으로 다시 부르면 같은 실패가 난다. 재시도마다 컨텍스트를 한 단계씩 강화한다(1차 실패 내용+수정 방향 → 2차 관련 스킬·정본 명시 지정 → 3차 유사 사례). **최대 3회**로 제한하고, 소진 시 태스크를 보류하고 PROGRESS에 실패 이력을 남긴 뒤 사람에게 넘긴다. 나머지 태스크는 계속 진행한다.
- 여러 세션에 걸친 반복 실패는 재시도 문제가 아니라는 진단 기준을 명시했다 — 계약 모호(→G 판정), 과제 과대(→분할), 팀 구성 부적합(→HE 재검토).
- `context-README.template.md`에 두 루프·승인 게이트·규칙 등급 섹션 추가.

## 0.3.0 (2026-07-23)

판정 체계를 정밀화했다. 내부 검토 자료(*Cloud Native 기반 AI Native의 진화*, 장덕성, 2026-04)의 Decay/Rot/Drift 분류, MUST/SHOULD Rule ID 체계, Observability≠Control 단일 책임 원칙을 CFHM의 계약 구조에 맞춰 반영했다.

- **D(점진 이탈) 판정 추가** — 기존 V/G/S로는 "개별 건은 위반이 아닌데 같은 방향으로 누적되는" 이탈을 잡지 못했다. 에러도 없고 테스트도 통과하므로 매 시점의 대조로는 영영 안 잡힌다. D는 한 회차가 아니라 **추세로 판정**하며, 정본에 감시 축과 임계를 미리 정의하고 QA가 회차 간 카운트를 누적한다.
- **규칙 등급과 Rule ID** — 계약 항목에 `[MUST]`(`{SEAM}-P###`, override 불가) / `[SHOULD]`(`{SEAM}-G###`, 이유 명시 시 가능) / 등급 없음(구현 자유)을 부여한다. 이전에는 정본의 모든 문장이 같은 무게여서 QA가 심각도를 매길 수 없었다. 심각도는 이제 등급에서 기계적으로 나온다 — MUST 위반=FAIL, SHOULD 이탈=WARN. Rule ID는 삭제 후에도 재사용하지 않는다(위반 이력 추적).
- **측정과 판정 분리** — 사실 수집 단계에서는 위반 여부를 말하지 않는다. 섞으면 "위반 같으니 그만 보자"며 수집을 중단하거나 "애매하니 넘어가자"며 사실을 누락시킨다.
- **`SENSOR_ERROR` 도입** — 검사 도구가 실행되지 않은 것과 이상이 없는 것을 구분한다. 미측정을 통과로 처리하면 아무도 보지 않은 곳이 검사를 통과한 것으로 둔갑한다. 기존 `SKIP`(대상 미존재)과도 분리했다.
- **미분류 관측** — 어떤 Rule과도 매칭되지 않는 사실은 판정하지 않고 G 후보로 기록한다. 규칙 추가가 필요하다는 신호다.
- `seam-contract.template.md` 재작성(등급·Rule ID·점진 이탈 감시·검증 항목·변경 이력), `seam-qa-report.template.md` 재작성(측정/판정 분리, D 누적 추적).
