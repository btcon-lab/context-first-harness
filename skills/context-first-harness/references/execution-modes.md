# 실행 모드와 오케스트레이션 (HE 상세)

에이전트를 어떤 방식으로 돌릴지(팀/서브/하이브리드), 산출물을 어떻게 주고받을지를 정한다. 이 결정은 오케스트레이터 스킬에 **명시적으로 기재**한다 — 적지 않으면 세션마다 다르게 실행된다.

## 모드 선택

| 모드 | 언제 | 수단 |
|------|------|------|
| **에이전트 팀** (기본) | 2+ 협업, 실시간 조율·피드백 교환, 중간 산출물 상호 참조 | `TeamCreate` + `SendMessage` + `TaskCreate`로 자체 조율 |
| **서브 에이전트** | 단일 작업, 결과만 반환하면 충분, 팀 통신 오버헤드가 이득보다 클 때 | `Agent` 직접 호출, `run_in_background`로 병렬 |
| **하이브리드** | Phase마다 성격이 다를 때 | Phase 단위로 혼합, 각 Phase에 모드 명시 |

**결정 순서:** ① 2명 이상이면 팀을 먼저 검토한다(기본값) → ② 통신이 구조적으로 불필요하고 결과 전달만이면 서브 → ③ Phase 성격이 확연히 다르면 하이브리드.

> **팀 도구를 쓸 수 없는 환경일 때:** 에이전트 팀은 실험 기능이라 활성화돼 있지 않을 수 있다(`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS`). `TeamCreate`가 없으면 **서브 에이전트 모드로 진행**하고, 오케스트레이터에 그 사실을 기록한다. 이때 계약 변경 통지는 메시지로 못 하므로 **정본을 갱신한 뒤 소비자 에이전트를 순차 재호출**해 반영시킨다 — 통지 단계를 건너뛰면 그대로 drift가 된다. CE-우선 프로토콜 자체는 모드와 무관하게 지킨다.

**CFHM에서 팀이 기본인 이유:** seam은 소유자 1명과 소비자 N명이 공유하는 사실이다. 소유자가 계약을 확정하면 소비자들이 즉시 알아야 하는데, 서브 에이전트는 결과가 메인에만 돌아가므로 이 통지가 한 홉 늦는다. 팀 모드의 `SendMessage`가 **CE-우선 프로토콜의 "통지" 단계와 그대로 대응**한다.

자주 쓰는 하이브리드 조합:
- 병렬 수집(서브) → 합의 통합(팀)
- 팀이 초안 생성 → 서브가 독립 검증
- Phase 간 팀 재구성: 산출물을 파일로 저장 → 팀 해체 → 새 팀

## 아키텍처 패턴

| 패턴 | 언제 | CFHM에서의 쓰임 |
|------|------|----------------|
| 파이프라인 | 순차 의존 | P-A → P-B → HE → P-C 본류 |
| 팬아웃/팬인 | 병렬 독립 → 통합 | seam별 계약 시드를 병렬로 |
| 전문가 풀 | 상황별 선택 호출 | seam 소유자 중 해당자만 호출 |
| 생성-검증 | 생성 후 품질 검수 | 구현 → Seam QA (점진 실행) |
| 감독자 | 중앙이 상태 관리·동적 분배 | 공유 작업목록 기반 |
| 계층 위임 | 재귀 분해 | 2단계 이내로 제한 |

## 에이전트 분리 기준

4축으로 판단한다. 하나라도 강하게 걸리면 분리한다.

| 축 | 분리 신호 |
|----|----------|
| **seam 오너십** | **서로 다른 seam을 소유한다 → 분리 (CFHM 1순위 기준)** |
| 전문성 | 요구되는 판단 기준이 다르다 |
| 병렬성 | 동시에 진행해도 무방하다 |
| 컨텍스트 | 필요한 참조 자산이 겹치지 않는다 |
| 재사용성 | 다른 하네스에서도 쓸 역할이다 |

역으로 **한 seam을 두 에이전트가 나눠 소유하면 안 된다.** SoT가 깨진다. 일이 많으면 소유자는 1명으로 두고 작업자를 붙인다.

## 데이터 전달 프로토콜

| 전략 | 방식 | 모드 | 적합 |
|------|------|------|------|
| 메시지 | `SendMessage` 직접 통신 | 팀 | 실시간 조율, **계약 변경 통지** |
| 태스크 | `TaskCreate`/`TaskUpdate` | 팀 | 진행 추적, 의존 관계, 작업 요청 |
| 파일 | 약속된 경로 read/write | 팀+서브 | 대용량, 구조화 산출물, 감사 추적 |
| 반환값 | `Agent` 반환 메시지 | 서브 | 메인이 결과 직접 수집 |
| **정본** | **`docs/context/{seam}.md`** | **전체** | **계약 사실 — 유일한 정답 경로** |

**권장 조합(팀):** 태스크(조율) + 파일(산출물) + 메시지(통지) + 정본(계약)
**권장 조합(서브):** 반환값(결과) + 파일(대용량) + 정본(계약)

> **계약 사실은 메시지나 반환값으로 전달하지 않는다.** 메시지로 흘린 계약은 어디에도 남지 않아 다음 세션에 소실되고, 받은 쪽이 자기 파일에 적으면 그게 곧 drift다. 계약은 **정본에 쓰고 위치만 통지**한다.

**파일 규칙:**
- 중간 산출물은 `_workspace/`에 저장
- 파일명: `{phase}_{agent}_{artifact}.{ext}` (예: `01_analyst_requirements.md`)
- 최종 산출물만 지정 경로로, `_workspace/`는 보존(사후 검증·감사 추적)
- **`_workspace/`는 정본이 아니다.** 확정된 계약 사실은 반드시 `docs/context/`로 승격한다.

## 팀 크기

| 작업 규모 | 팀원 | 팀원당 작업 |
|----------|------|-----------|
| 소규모(5~10 작업) | 2~3명 | 3~5 |
| 중규모(10~20) | 3~5명 | 4~6 |
| 대규모(20+) | 5~7명 | 4~5 |

팀원이 늘수록 조율 오버헤드가 커진다. 집중된 3명이 산만한 5명보다 낫다. **seam 수가 팀원 수의 하한**이 되지만(각 seam에 소유자가 필요), seam이 많으면 한 에이전트가 인접 seam 여러 개를 소유해도 된다 — 소유자가 1명이기만 하면 SoT는 지켜진다.

## 재시도 — 같은 조건으로 다시 부르지 않는다

에이전트가 실패했는데 **똑같은 입력으로 다시 부르면 똑같이 실패한다.** 재시도가 의미를 가지려면 매 회차 조건이 달라져야 한다. 그래서 재시도마다 컨텍스트를 한 단계씩 더 준다.

| 회차 | 추가로 주는 것 | 노림수 |
|------|---------------|--------|
| 1차 | 실패 내용 + 수정 방향(무엇이 왜 틀렸는가) | 단순 실수면 여기서 끝난다 |
| 2차 | 1차 + 관련 스킬·정본을 명시적으로 지정해 로드 | 필요한 지식을 못 찾은 경우 |
| 3차 | 2차 + 유사 사례(이미 통과한 다른 seam의 구현) | 추상적 지시를 못 옮기는 경우 |
| 소진 | — | **사람에게 넘긴다** |

**3회로 제한한다.** 4차 이후에도 안 되는 건 컨텍스트 부족이 아니라 과제 정의나 계약 자체의 문제일 가능성이 높다. 무한 재시도는 토큰만 태우고 같은 실패를 반복한다.

**소진 시 처리** — 해당 태스크를 보류로 두고, `PROGRESS.md`에 실패 이력을 남긴 뒤 사람에게 알린다. 나머지 태스크는 계속 진행한다(한 건 때문에 전체를 멈추지 않는다). 보고서에 누락을 명시한다.

```
## 실패 이력
- 태스크: {무엇}
- 위반/실패: {Rule ID 또는 증상}
- 재시도: 3회 소진
- 원인 추정: {계약이 모호한가, 과제가 큰가, 도구가 없는가}
- 필요 조치: {사람이 판단할 것}
```

**같은 태스크가 여러 세션에 걸쳐 반복 실패하면 재시도 문제가 아니다.** 계약이 모호하거나(→ G 판정, 정본 확정), 과제가 너무 크거나(→ 분할), 에이전트 구성이 안 맞는 것이다(→ HE 재검토).

## 에러 핸들링

오케스트레이터에 방침을 명시한다.

| 상황 | 처리 |
|------|------|
| 에이전트 실패 | 위 재시도 정책(최대 3회, 매 회차 컨텍스트 강화) → 소진 시 보류 + 사람 알림 |
| 상충 데이터 | 삭제하지 않고 **출처 병기** 후 판단 유보 |
| 계약 위반 산출물 | 정본 기준으로 반려(구현 수정). 계약을 산출물에 맞추지 않는다 |
| 정본에 `[TODO]` 남음 | 진행 차단이 아니라 `open-decisions.md` 등재 후 진행, QA에서 G 후보로 |
| 팀원 무응답 | 태스크 상태로 감지, 리더가 재할당 |

## 연기된 운영 결정 게이트 (Phase 0)

P-F에서 "연기"로 남긴 결정(제어 루프 등)은 **조건이 충족되는 순간 자동으로 실행되지 않는다.** 다시 수면 위로 올려주는 장치가 없으면, 남긴 결정도 결국 잊힌다 — 조용한 스킵과 다를 바 없어진다.

그래서 오케스트레이터는 **매 실행의 Phase 0에서 PROGRESS의 "하네스 운영 결정"을 읽고**, 연기된 항목의 실행 조건이 지금 충족됐는지 확인한다.

```
Phase 0: PROGRESS "하네스 운영 결정" 로드
   → 연기된 결정이 있는가?
      → 실행 조건 충족? (예: 제어 루프 "M3 착수 시" ↔ 지금 구현 단계 진입)
         → 예 → 아래 "실행 스캐폴딩"을 이번 실행의 선행 작업으로 올린다
         → 아니오 → 그대로 두고 진행
```

**중요: 조건 판정은 사람에게 확인받는다.** "지금이 M3인가"는 오케스트레이터가 단정하지 않고 사용자에게 묻는다. 판단을 없애는 게 아니라, 잊히지 않게 올려주는 것이 이 게이트의 목적이다.

### 이 게이트는 생성되는 오케스트레이터 스킬 본문에 **복제**한다

여기(메타 스킬 reference)에 있는 것만으로는 부족하다. **실제 개발 단계에서 사용자는 CFHM을 다시 돌리지 않고, 생성된 오케스트레이터 스킬("다음 단계 진행")을 직접 호출한다.** 그 오케스트레이터의 Phase 0에 게이트가 박혀 있지 않으면, 정작 조건이 충족되는 구현 단계(M3 등)에 게이트가 돌지 않는다 — 연기 결정이 그때 조용히 잊힌다.

그러므로 HE에서 오케스트레이터 스킬을 만들 때, **그 스킬의 Phase 0(컨텍스트 확인) 안에 이 게이트를 명시적으로 포함**한다. 초기/부분/CE 재실행 판별과 **나란히**, 다음을 오케스트레이터 본문에 적는다:

```markdown
## Phase 0: 컨텍스트 확인 (매 실행 시작)
...초기/부분/CE 재실행 판별...

### 연기된 운영 결정 확인
- `PROGRESS.md`의 "하네스 운영 결정"을 읽는다.
- 연기된 항목(제어 루프 등)의 실행 조건이 지금 충족됐는지 본다.
  - 예: 제어 루프 "M3 착수 시" ↔ 지금 구현 단계 진입인가?
- 충족 판정은 **사용자에게 확인**받는다. 단정하지 않는다.
- 충족 시: 시드 도출(seam·rules 자동, command는 사람) 후 설치 —
  절차 `context-first-harness/references/control-loop.md`.
- 미충족 시: 그대로 두고 진행. **평가했다는 사실을 PROGRESS 세션 로그에 남긴다**
  (조건 대조 결과가 기록되어야 "매 실행 재부상"이 검증된다).
```

메타 스킬 reference는 "어떻게 만드는지"의 근거이고, **실제로 매 실행 도는 것은 생성된 오케스트레이터**다. 게이트는 후자에 있어야 값을 한다.

### 실행 스캐폴딩 — 무엇을 자동으로, 무엇을 사람이

연기된 제어 루프를 실제로 설치할 때, **정본에서 도출 가능한 것은 자동으로 채우고, 프로젝트 고유값만 사람에게 남긴다.**

| 센서 필드 | 출처 | 채우는 주체 |
|-----------|------|------------|
| `seam` | 정본 파일명 (`api-contract.md` → `api`) | **자동** |
| `rules` | 정본의 Rule ID 목록 | **자동** |
| `match` | 코드 경로 — `docs/design`에 모듈 경로가 정해졌으면 도출 | 반자동 |
| `command` | 프로젝트 실제 검증 명령 | **사람 / `[TODO]`** |

**`command`를 추측해서 채우지 않는다.** 존재하지 않는 명령을 넣으면 `SENSOR_ERROR`를 양산하고, 그것을 통과로 처리하면 v0.3.0의 "미측정을 정상으로 읽지 않는다" 원칙이 깨진다. 자동 설치는 "판단 없는 실행"이 되어, 조용한 스킵의 거울상이 된다. `command`가 빌 때는 `[TODO]`로 두고 그 센서는 아직 돌지 않게 한다.

절차 상세: `references/control-loop.md`의 "정본에서 센서 시드 도출".
