# 스킬 테스트 (Phase 8 상세)

생성된 스킬은 **실제로 실행해 봐야** 검증된다. 파일 존재 확인과 frontmatter 점검은 구조 검증일 뿐, 스킬이 작동하는지는 말해주지 않는다.

## 1. 트리거 검증

스킬이 아무리 좋아도 트리거되지 않으면 없는 것과 같다. description만으로 판정되므로 별도 검증이 필요하다.

### should-trigger (8~10개)
트리거되어야 하는 표현을 다양하게 만든다. 한 가지 어투만 쓰면 검증이 되지 않는다.
- 공식적 / 캐주얼 — "API 계약을 갱신해줘" / "응답 형식 좀 바꿔줘"
- 명시적 / 암시적 — "seam 계약 수정" / "프론트가 쓰는 필드 이름 바꿔야 해"
- **후속 요청** — "아까 그거 다시", "계약 부분만 업데이트"
- 도메인 용어 / 일상어 — "RBAC 정본" / "누가 뭘 볼 수 있는지 정리"

### should-NOT-trigger (8~10개, near-miss)
**여기가 실질이다.** "피보나치 함수 작성"처럼 명백히 무관한 쿼리는 테스트 가치가 0이다. 키워드는 겹치지만 다른 스킬이 맡아야 하는 **경계가 모호한 쿼리**를 만든다.

| 유형 | 예 (api-contract 스킬 기준) |
|------|---------------------------|
| 인접 seam | "권한 규칙 바꿔줘" → rbac 스킬 |
| 계약 vs 구현 | "이 엔드포인트 구현해줘" → 구현 에이전트 |
| 계약 vs 검증 | "API랑 프론트 안 맞는지 봐줘" → seam-qa |
| 정본 vs 원문 | "계획서에 API 뭐라고 써 있어?" → plan-extracts 조회 |
| 전역 vs 개별 | "하네스 전체 점검" → 오케스트레이터 |

CFHM은 seam 단위로 스킬이 쪼개지므로 **인접 seam 간 오트리거가 가장 흔한 실패**다. near-miss의 절반 이상을 인접 seam으로 채운다.

### 충돌 점검
기존 스킬(다른 플러그인 포함)과 트리거가 겹치는지 확인한다. 겹치면 양쪽 description에 경계 문장을 넣는다.

## 2. 실행 테스트

각 스킬에 대해 현실적인 테스트 프롬프트 2~3개를 만든다. 실제 사용자가 칠 법한 구체적이고 자연스러운 문장이어야 한다 — 스킬 문구를 그대로 옮긴 프롬프트는 통과가 보장돼 검증이 되지 않는다.

### with-skill vs without-skill 비교
가능하면 에이전트를 두 개 띄워 같은 프롬프트를 실행한다.
- **with-skill:** 스킬을 읽고 수행
- **without-skill(baseline):** 스킬 없이 수행

차이가 없으면 그 스킬은 무게를 벌지 못하는 것이다. 삭제하거나 내용을 강화한다. 하네스가 비대해지는 가장 흔한 원인이 **아무도 검증하지 않은 무가치 스킬**이다.

### 평가
| 성질 | 방법 |
|------|------|
| 객관 검증 가능(파일 생성, 필드 추출, 형식 준수) | assertion 정의 후 기계적 판정 |
| 주관(문체, 구성, 설득력) | 사용자 리뷰 |
| **CFHM 고유** | **정본 준수 여부** — 산출물이 계약을 지켰는지 Seam QA 기준으로 판정 |

CFHM은 정본이 있으므로 "좋아 보인다"에 의존하지 않고 계약 준수를 assertion으로 쓸 수 있다. 실행 테스트의 평가 기준으로 `references/seam-qa.md`의 assertion 전개를 재사용한다.

### 반복 개선
문제가 나오면:
1. 피드백을 **일반화**해 스킬을 고친다 — 그 테스트 케이스만 통과시키는 수정은 다음 입력에서 깨진다
2. 재테스트
3. 의미 있는 개선이 더 없을 때까지 반복
4. 여러 테스트에서 공통으로 작성되는 코드가 보이면 `scripts/`에 번들링

## 3. 드라이런

실행 없이 설계를 훑는다.
- 오케스트레이터의 Phase 순서가 논리적인가
- 데이터 전달 경로에 dead link(쓰는 쪽만 있고 읽는 쪽이 없는 파일)가 없는가
- 각 에이전트의 입력이 이전 Phase의 출력과 매칭되는가
- 에러 시나리오별 폴백이 실제로 실행 가능한가
- **모든 seam에 소유자가 정확히 1명인가, 소비자가 0명인 seam은 없는가** (0명이면 seam이 아니라 design이다 → P-B 재검토)

## 4. 테스트 시나리오 기록

오케스트레이터 스킬에 `## 테스트 시나리오` 섹션을 남긴다. 정상 흐름 1개 + 에러 흐름 1개 이상. 다음 세션에서 회귀 확인의 기준이 된다.

## 검증 순서

```
구조 검증 → 정본 무결성 → 트리거 검증 → 드라이런 → 실행 테스트 → Seam QA
(싼 것 → 비싼 것)
```
구조가 깨졌는데 실행 테스트를 돌리는 것은 낭비다. 싼 검증부터 통과시킨다.
