# 스킬 작성 (HE 상세)

에이전트는 "누가", 스킬은 "어떻게", 정본은 "무엇". 스킬은 절차 지식을 담고, 사실은 담지 않는다 — **사실을 스킬에 적는 순간 정본과의 drift가 시작된다.**

## 구조
```
skill-name/
├── SKILL.md          필수 — frontmatter(name, description) + 본문
├── references/       조건부 로딩 문서 (무제한)
├── assets/           출력에 쓰는 템플릿·파일
└── scripts/          반복·결정적 작업 실행 코드
```

## description — 유일한 트리거 메커니즘

description은 본문과 달리 **항상 컨텍스트에 있고**, 스킬이 발동할지를 혼자 결정한다. 트리거 판단은 보수적으로 일어나므로 소극적으로 쓰면 있어도 안 쓰인다.

**나쁨:** `"API 문서를 관리하는 스킬"`
**좋음:** `"API 계약 정본(docs/context/api.md)을 작성·갱신한다. 엔드포인트 추가/응답 형식 변경/에러 코드 정의, '계약 수정', 'API 스펙', '응답 바꿔줘' 요청 시 사용. 구현 코드 작성은 대상이 아님."`

담을 것 네 가지:
1. **무엇을 하는가** — 구체적 동작 나열
2. **언제 트리거되는가** — 실제 사용자 표현으로
3. **후속 작업 키워드** — "다시 실행", "재실행", "업데이트", "수정", "보완", "이전 결과 기반으로". 초기 생성 키워드만 넣으면 두 번째 요청부터 안 잡힌다.
4. **트리거되면 안 되는 경계** — 유사하지만 다른 스킬이 맡을 일

CFHM 스킬은 여기에 **담당 seam을 명시**한다. 어떤 정본을 소유/참조하는지가 description에 있으면 오케스트레이터가 배선을 검증할 수 있다.

## 본문 원칙

| 원칙 | 이유 |
|------|------|
| **Why를 함께 쓴다** | "금지"만 적으면 엣지 케이스에서 오판한다. 이유를 알면 응용된다. |
| **Lean하게** | 컨텍스트는 공공재. 500줄 이내, 초과분은 references/로. |
| **일반화한다** | 특정 예시에 맞춘 좁은 규칙은 다른 입력에서 깨진다(오버피팅). |
| **명령형으로** | "~한다", "~하라". 설명체는 지시로 읽히지 않는다. |
| **반복 코드는 번들링** | 매번 같은 스크립트를 짜게 되면 `scripts/`에 미리 넣는다. |
| **사실은 정본에 둔다** | 스킬에는 절차만. 엔드포인트·필드명·상태값을 본문에 쓰면 계약 변경 시 두 곳을 고쳐야 한다. |

## Progressive Disclosure

| 단계 | 로딩 시점 | 크기 목표 |
|------|----------|----------|
| Metadata(name+description) | 항상 | ~100단어 |
| SKILL.md 본문 | 트리거 시 | <500줄 |
| references/ | 필요할 때만 | 무제한 |
| **정본(docs/context)** | **Read로 필요 시** | 무제한 |

CFHM은 4단계다. 정본을 스킬 본문에 복사하거나 "항상 로드"로 지정하지 않는다 — 그 순간 SoT가 깨지고 컨텍스트가 부푼다.

**분리 규칙:** 본문이 500줄에 근접하면 세부를 references/로 옮기고 "언제 이 파일을 읽으라"는 포인터만 남긴다. 300줄 넘는 reference에는 상단에 목차를 넣는다. 도메인별 변형은 파일을 나눠 해당 것만 로드되게 한다.

## 컨텍스트 자산 참조 섹션 (CFHM 필수)

모든 스킬 본문에 이 섹션을 넣는다. P-D 배선의 산출물이며, 이게 없으면 스킬이 정본을 안 읽고 자기가 기억하는 사실로 작업한다.

```markdown
## 컨텍스트 자산 참조
- **소유 정본:** `docs/context/{seam}.md` — 이 스킬이 갱신 권한을 가진다. 변경 시 CE-우선 프로토콜(정본 먼저 → 통지 → 반영 → 검증 → 기록)을 따른다.
- **참조 정본:** `docs/context/{other}.md` — Read로 로드. **복제·재기술 금지, 링크만.**
- **용어:** `docs/context/glossary.md`
- 계약과 구현이 어긋나면 **정본이 진실**이다. 임의 수정 대신 소유 에이전트에게 통지한다.
```

## 스킬-에이전트-seam 연결

- 에이전트 1 ↔ 스킬 1~N
- 여러 에이전트가 공유하는 스킬 가능
- **정본 1 ↔ 소유 스킬 1** — 한 정본을 두 스킬이 갱신하면 SoT가 깨진다. 참조는 N개 무방.

## 중복 검토 (신규 생성 전 필수)

하네스를 반복 구축하면 이름만 다른 유사 스킬이 쌓인다. 생성 전 `대상/.claude/skills/`를 확인한다.

| 상황 | 조치 |
|------|------|
| 기능이 거의 같음 | 기존 스킬 재사용, 생성하지 않음 |
| 절차는 같고 대상 seam만 다름 | 기존 스킬을 **일반화**하고 대상을 파라미터로 |
| 일부 겹침 | 겹치는 부분을 references/로 분리해 공유 |
| 완전히 다름 | 신규 생성 |

## 안티패턴
- description에 후속 키워드 누락 → 재실행 요청이 안 잡힌다
- 정본 내용을 본문에 복사 → drift 발생
- 500줄 초과 방치 → 트리거될 때마다 컨텍스트 낭비
- "컨텍스트 자산 참조" 섹션 없음 → 스킬이 정본과 무관하게 동작
- 커맨드(`.claude/commands/`) 생성 → CFHM은 만들지 않는다
