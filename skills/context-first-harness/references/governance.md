# P-E: 거버넌스 (상세)

`대상/docs/context/README.md`(템플릿 `assets/context-README.template.md`)를 만들어 정본 운영 규칙을 고정한다. 이것이 없으면 정본이 "그냥 문서"로 전락하고 drift가 돌아온다.

## 담아야 할 4가지
0. **규칙 등급 체계** — 계약 항목은 `[MUST]`(정책, override 불가, `{SEAM}-P###`) / `[SHOULD]`(지침, 이유 명시 시 override 가능, `{SEAM}-G###`) / 등급 없음(구현 자유)으로 나눈다. 등급이 없으면 모든 문장이 같은 무게가 되어 **QA가 심각도를 매길 수 없고**, ID가 없으면 위반 이력이 규칙 단위로 쌓이지 않는다. Rule ID는 삭제 후에도 재사용하지 않는다(과거 이력 추적).
1. **SoT 규칙** — 같은 사실 한 곳, 정본을 읽어 참조·복제 금지, 정본↔검증, 추적성.
2. **Context(seam) vs Design 경계** — 무엇이 context(공유 사실), 무엇이 design(단일 소유 구현). 판단 기준: "2+ 소비자 공유 사실인가?"
3. **CE-우선 변경 프로토콜** — 계약 변경은 **정본 먼저** → 통지 → 반영 → 검증 → 기록. Drift 시 정본이 진실, 정본 우회 변경 반려. 순수 구현 세부는 예외.
4. **변경 이력 테이블** — 계약이 바뀔 때마다 한 줄(날짜·정본·내용·사유·통지).

## 왜 CE-우선인가
코드/화면을 먼저 바꾸고 정본을 나중에 맞추면, 정본이 "뒤따라가는 문서"가 되어 신뢰를 잃는다. 정본이 항상 앞서고 나머지가 따라와야 단일 진실원천이 유지된다.

## 두 개의 루프 — 고치는 것과 배우는 것

검증에서 문제가 나오면 할 일이 두 갈래로 갈린다. **이 둘을 한 흐름으로 처리하면 계약이 조용히 침식된다.**

| | 즉시 교정 (Runtime) | 계약 개선 (Learning) |
|---|---|---|
| 무엇을 고치나 | **구현** | **정본** |
| 언제 | 판정이 V(구현 위반)일 때 | 판정이 G·S·D일 때 |
| 속도 | 즉시, 자동 | 느리게, 검토 후 |
| 승인 | 불필요 — 계약이 이미 정답 | **사람 승인 필수** |
| 남는 기록 | 수정 커밋 | 변경 이력 + PROGRESS |

교정 루프는 계약을 정답으로 두고 구현을 맞추는 일이라 자동으로 돌아도 안전하다. 학습 루프는 **정답 자체를 바꾸는 일**이라 다르다.

## 정본 변경에는 사람 승인이 필요하다

계약은 아키텍처 결정이다. 에이전트가 검증 결과만 보고 정본을 고치면, 당장은 QA가 통과하지만 **"구현에 맞춰 계약을 낮추는" 방향으로 흘러간다.** 몇 회차만 반복돼도 정본은 코드의 사후 요약으로 전락하고, 애초에 정본을 둔 이유가 사라진다.

그래서 흐름을 이렇게 나눈다.

```
검증에서 G·S·D 판정
   ↓
QA가 개정을 [제안]              ← 직접 고치지 않는다
   ↓
소유 에이전트가 초안 작성        ← 여전히 제안 단계
   ↓
★ 사람 검토·승인 ★             ← 여기가 게이트
   ↓
정본 반영 + 변경 이력 기록
   ↓
소비자 전원에게 통지 → 반영 → 재검증
```

**제안에 담을 것** — 무엇을(대상 정본·Rule ID), 어떻게(현재 → 제안), 왜(판정 근거), 영향 범위(이 seam의 소비자 목록). 이게 없으면 사람이 판단할 수 없어 승인이 형식적으로 변한다.

**승인 없이 진행 가능한 예외** — `[TODO]`를 처음 채우는 경우(기존 결정을 뒤집지 않으므로), 오타·서식 교정, 링크 갱신. 그 외 규칙의 추가·완화·삭제는 전부 승인 대상이다.

**규칙 완화(MUST → SHOULD 또는 삭제)는 가장 엄격하게 본다.** 반복 위반이 쌓였다는 건 규칙이 틀렸다는 뜻일 수도, 팀이 못 지키고 있다는 뜻일 수도 있다. 전자면 완화가 맞고 후자면 완화는 문제를 덮는 것이다. **판단은 사람이 한다.**

## 오너십 맵
seam → 소유 에이전트(1명) → 소비 에이전트(N명)를 README나 별도 표로 남긴다. 소유자만 해당 정본을 **변경 제안**을 할 수 있고, 승인 후 반영하며, 변경 시 소비자에게 통지한다.

## 판단 훈련 (경계 예시)
| 변경 | context? design? |
|------|------------------|
| 응답 필드 하나 추가/이름변경 | **context**(계약) — 소비자 영향 |
| 내부 함수 리팩터링 | design(구현 세부) — 예외 |
| 상태 하나 추가 | **context**(전이 맵) |
| 로깅·주석·테스트 추가 | design/구현 — 예외 |
| 권한 규칙 변경 | **context**(RBAC 정본) |
