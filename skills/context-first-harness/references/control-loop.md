# 제어 루프 — 실행을 관측하고 교정한다

계약을 세우고 팀을 만들어도, **작업이 진행되는 동안 계약이 지켜지는지 아무도 안 보면** 이탈은 통합 시점에야 드러난다. 그때는 이미 오염이 퍼진 뒤다.

## 쓸지 말지 — 결정을 남긴다

제어 루프는 선택 기능이다. 하지만 **"옵션이니 안 만든다"고 조용히 지나가지 않는다.** 그렇게 스킵하면, 정작 이득이 큰 프로젝트에서도 아무도 검토하지 않은 채 빠진다. 셋 중 하나로 결정하고 그 판단을 PROGRESS에 남긴다.

| 결정 | 언제 | 남기는 것 |
|------|------|----------|
| **설치** | seam이 여럿(대략 4개+)이거나 장기 구현 — 이탈을 조기에 잡는 이득이 크다 | `seam-sensors.json` + 훅 설치, 무엇을 감시하는지 |
| **연기** | 하네스 구성만 끝났고 구현 착수 전이라 **검사할 코드가 아직 없다** | "구현 단계에서 제어 루프 설치"를 PROGRESS 다음 할 일로 등재 |
| **생략** | seam이 적고 단기 작업이라 과하다 | 생략 사유 한 줄 |

**하네스를 막 구성한 시점은 대개 "연기"다.** 센서는 구현된 코드를 검사하는 것이라, 코드가 없으면 돌 게 없다. 이때 "그래서 안 만든다"가 아니라 **"구현 시작할 때 만든다"를 명시적 할 일로 남기는 것**이 핵심이다 — 그래야 구현 단계에서 잊히지 않는다.

제어 루프는 이 감시를 자동화한다. 네 개 층이 순서대로 돌고, 마지막에서 다시 처음으로 돌아온다.

```
실행 ──▶ 관측 ──▶ 판정 ──▶ 교정 ──┐
 ▲                              │
 └──────────────────────────────┘
                    └──▶ 계약 개선(사람 승인) ──▶ 정본
```

## 각 층은 하나만 한다

| 층 | 책임 | 입력 | 출력 | 하지 않는 것 |
|----|------|------|------|-------------|
| **실행** | 일을 한다 | 태스크 + 주입된 컨텍스트 | 파일 변경 | 자기 결과를 판정하지 않는다 |
| **관측** | 사실을 모은다 | 파일 변경 | 관측 리포트(사실) | **"위반인가"를 말하지 않는다** |
| **판정** | 정본과 대조한다 | 관측 리포트 | V/G/S/D + FAIL/WARN + Rule ID | 코드를 새로 읽지 않는다 |
| **교정** | 고치거나 배운다 | 판정 | 수정 지시 / 계약 개정 제안 | 정본을 직접 고치지 않는다 |

**관측과 판정을 분리하는 이유**는 `seam-qa.md`와 같다. 재는 쪽이 판단까지 하면 "위반 같으니 그만 보자"며 수집을 중단하거나 "애매하니 넘어가자"며 사실을 누락시킨다.

**상태를 바꾸는 층은 실행뿐이다.** 관측·판정은 읽기만 하고, 교정은 실행에 지시를 돌려준다. 이 제약이 있어야 루프를 몇 번 돌려도 부작용이 쌓이지 않는다.

## 층을 잇는 것은 훅이다

층끼리 직접 호출하지 않는다. **사건이 나면 훅이 다음 층을 깨우고, 결과를 함께 넘긴다.**

| 전이 | 언제 | 무엇을 넘기나 | 수단 |
|------|------|--------------|------|
| 실행 → 관측 | 파일이 바뀜 | 바뀐 파일 목록 | PostToolUse 훅 (`seam-sensor-run.sh`) |
| 관측 → 판정 | 센서 실행 완료 | 관측 리포트 | 리포트 파일 |
| 판정 → 교정 | FAIL 또는 WARN | 판정 + Rule ID + 근거 | Claude의 판단 |
| 교정 → 실행 | 수정 지시 생성 | 기존 컨텍스트 + 수정 방향 | 재시도(`execution-modes.md`) |
| 교정 → 정본 | 계약 문제로 분류 | 개정 **제안** | **사람 승인**(`governance.md`) |

직접 호출을 없애면 한 층이 실패해도 나머지가 멈추지 않는다. 센서 하나가 죽어도 다른 센서는 돌고, 결과는 `SENSOR_ERROR`로 남는다.

## 센서 — 무엇을 어떻게 재나

센서는 "코드가 바뀌면 자동으로 도는 검사"다. 대상 프로젝트의 `.claude/seam-sensors.json`에 정의한다(서식: `assets/seam-sensors.template.json`).

```json
{
  "id": "api-contract",
  "seam": "api",
  "rules": ["API-P001"],
  "match": ["src/api/**"],
  "command": "npm run test:contract",
  "measures": "API 응답 형식이 계약과 같은 모양인가",
  "timeout": 120
}
```

**설계 규칙:**

1. **센서는 seam에 매인다.** `seam`과 `rules`를 적어두면 판정할 때 어떤 정본·어떤 규칙과 대조할지가 바로 나온다. 이게 없으면 결과만 있고 대조 기준이 없다.
2. **변경 범위로 선택 실행한다.** 매번 전부 돌리면 느려서 아무도 안 쓴다. `match`에 걸린 것만 돈다.
3. **측정만 한다.** 센서의 종료코드와 출력은 **사실**이다. "종료코드 1 = 위반"이 아니다 — 테스트 환경 문제일 수도 있다. 판정은 정본과 대조해서 한다.
4. **실패해도 멈추지 않는다.** 센서가 못 돌면 `SENSOR_ERROR`로 기록하고 다음 센서로 넘어간다. 단 **통과로 처리하지 않는다.**
5. **타임아웃을 둔다.** 검사가 세션을 붙잡으면 훅 자체가 방해물이 된다.

무엇을 센서로 만들지는 seam 유형을 따른다 — 계약 테스트(API), 상태 전이 검사(상태머신), 타입 검사(데이터모델), 권한 매트릭스 대조(RBAC). 착안점은 `seam-qa.md`의 "경계면 버그 패턴".

## 에스컬레이션 — 루프가 안 끝날 때

| 상황 | 처리 |
|------|------|
| 센서가 3회 연속 `SENSOR_ERROR` | 검사 체계 자체의 문제다. 사람에게 알린다 |
| 재시도 3회 소진 | 태스크 보류 + PROGRESS에 실패 이력 (`execution-modes.md`) |
| 어떤 Rule과도 매칭 안 되는 관측 | 판정하지 말고 **G 후보**로 기록. 계약에 규칙이 없다는 뜻 |
| 같은 Rule ID에서 위반 반복 | 규칙 자체를 볼 때다. **완화는 사람이 판단한다** (`governance.md`) |
| 판정이 계속 FAIL | 계약이 모호하거나 과제가 크다. 재시도로 풀지 않는다 |

**루프가 스스로 못 빠져나오는 상태를 방치하지 않는다.** 자동화의 목적은 사람을 빼는 게 아니라, 사람이 봐야 할 것만 남기는 것이다.

## 정본에서 센서 시드 도출 (연기 → 설치 전환 시)

P-F에서 "연기"로 남긴 제어 루프를 구현 단계에서 실제로 설치할 때, **센서 설정을 백지에서 쓰지 않는다.** 대부분은 정본에 이미 있다. 정본에서 도출 가능한 것은 자동으로 채우고, 프로젝트 고유값만 사람에게 남긴다.

| 센서 필드 | 어디서 오나 | 채우는 주체 |
|-----------|-----------|------------|
| `id` | seam 이름 + 검사 종류 (`api-contract`) | 자동 |
| `seam` | 정본 파일명 (`api-contract.md` → `api`) | 자동 |
| `rules` | 정본의 Rule ID 목록 (특히 불변 규칙·검증 항목의 대상 ID) | 자동 |
| `measures` | 정본 "검증 항목"의 assertion 한 줄 | 자동 |
| `match` | 코드 경로 — `docs/design`에 모듈 경로가 정해졌으면 도출, 아니면 사람 | 반자동 |
| `command` | 프로젝트 실제 검증 명령 | **사람 / `[TODO]`** |
| `timeout` | 기본값 120 | 자동 |

**도출 절차:**
1. 오너십 맵(`docs/context/README.md`)에서 seam 목록을 읽는다.
2. seam마다 정본을 열어 "검증 항목" 표의 assertion과 대상 Rule ID를 모은다. 이게 `rules`와 `measures`가 된다.
3. `match`는 설계 문서에 모듈 경로가 있으면 채우고, 없으면 사람에게 묻는다.
4. **`command`는 빈 문자열(`""`)로 둔다.** 실제 검증 명령(계약 테스트·타입 검사·전이 검사 등)은 구현자가 채운다. 미확정 표시는 `_todo` 같은 별도 필드에 남긴다 — `command`에 `[TODO]` 같은 문자열을 넣으면 훅이 그것을 명령으로 실행해 버린다. `command`가 비면 훅은 그 센서를 건너뛴다(실행할 게 없으므로).

**`command`를 추측해서 채우지 않는 이유.** 존재하지 않는 명령을 넣으면 훅이 그것을 실행해 `SENSOR_ERROR`를 양산한다. 그리고 그 `SENSOR_ERROR`를 통과로 처리하면 "미측정을 정상으로 읽지 않는다"(가장 위험한 오류)가 깨진다. **자동으로 command를 지어내는 것은 "판단 없는 설치"이며, 조용한 스킵의 거울상이다** — 안 만드는 것도, 아무렇게나 만드는 것도 둘 다 판단이 빠진 것이다. command가 빈 센서는 설정에 남겨두되 그 항목은 아직 돌지 않게 한다(`command` 없으면 훅이 건너뛴다).

이 도출은 오케스트레이터가 구현 단계 진입 게이트에서 수행한다: `execution-modes.md`의 "연기된 운영 결정 게이트".

## 설치

1. 스크립트를 대상에 복사하고 실행 권한을 준다.
   ```
   mkdir -p 대상/.claude/hooks
   cp skills/context-first-harness/scripts/seam-sensor-run.sh 대상/.claude/hooks/
   chmod +x 대상/.claude/hooks/seam-sensor-run.sh
   ```
2. 센서 설정을 만든다 — `assets/seam-sensors.template.json`을 `대상/.claude/seam-sensors.json`으로 복사해 seam에 맞게 고친다.
3. `assets/settings-hooks.snippet.json`의 `PostToolUse` 블록을 대상 설정에 **병합**한다(기존 설정 보존).
4. `.gitignore`에 `_workspace/`를 넣는다(관측 리포트가 쌓인다).

**주의**

- 센서 설정 파싱에 `jq`가 필요하다. 없으면 이 훅은 조용히 건너뛴다(다른 기능은 정상).
- `command`에 적은 것은 훅이 **그대로 실행한다.** 직접 작성한 명령만 넣는다.
- 세션 중에 훅을 처음 설치하면 로드되지 않을 수 있다. `/hooks`를 한 번 열거나 새 세션을 연다.
- 센서 설정이 없으면 훅은 아무것도 하지 않는다. **제어 루프는 선택 기능이다** — seam이 적은 프로젝트에는 과하다.

## 검증

```
# 설정 없음 → 조용히 통과
echo '{"tool_input":{"file_path":"x.ts"}}' | 대상/.claude/hooks/seam-sensor-run.sh; echo $?   # 0

# 매칭되는 파일 → 관측 리포트 생성
echo '{"tool_input":{"file_path":"src/api/jobs.ts"}}' | CLAUDE_PROJECT_DIR=대상 대상/.claude/hooks/seam-sensor-run.sh
ls 대상/_workspace/observations/
```

무관한 파일을 넣었을 때 리포트가 **안 만들어져야** 정상이다. 만들어진다면 `match` 패턴이 너무 넓다.
