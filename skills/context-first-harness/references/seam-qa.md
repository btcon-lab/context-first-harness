# Seam QA — 경계면 계약 대조 검증

경계면 버그는 "같은 사실을 양쪽이 다르게 알고 있을 때" 난다. 단위 테스트는 각 쪽이 **자기가 믿는 사실**에 대해 통과하므로 이 버그를 잡지 못한다. 그래서 QA의 단위는 파일이나 함수가 아니라 **seam**이다.

## 왜 CFHM의 QA가 다른가 (계약이 정답지다)

계약 없는 하네스의 QA는 **코드 ↔ 코드** 대조다. 이 방식은 두 가지 한계가 있다.

1. **대칭적이라 판정이 안 된다.** API가 `snake_case`, 프론트가 `camelCase`면 불일치는 알아도 **어느 쪽이 틀렸는지** QA가 말할 수 없다. 결국 사람에게 "어느 쪽이 맞나요?"를 되묻는다.
2. **검사 대상을 탐색해야 한다.** 어디가 경계면인지 코드베이스를 뒤져 추론해야 하고, 놓친 경계면은 영원히 검사되지 않는다.

CFHM은 **정본 ↔ 구현** 대조다. 비대칭이므로 판정이 나오고, P-B의 오너십 맵이 검사 대상을 미리 준다.

| | 계약 없는 QA | Seam QA (CFHM) |
|---|---|---|
| 대조축 | 구현 A ↔ 구현 B | **정본 ↔ 구현 N개** |
| 판정 | 불가(대칭) — "다름"까지만 | **가능(비대칭)** — 정본이 진실 |
| 검사 대상 | 코드에서 추론·탐색 | **오너십 맵이 제공** |
| 검사 항목 | 사람이 착안 | **계약 항목이 곧 assertion** |
| 누락 감지 | 어려움 | 소비자 목록 대조로 기계적 |

> **불변 규칙: 정본이 진실이다.** 구현이 정본과 다르면 기본 판정은 "구현 위반"이다. 정본이 현실을 못 담은 경우라도 코드를 정답으로 승격하지 않고, **CE-우선 변경 프로토콜**(정본 먼저 개정 → 통지 → 반영)을 태운다.

## 측정과 판정을 분리한다

QA는 두 가지 일을 하는데, **섞으면 둘 다 나빠진다.**

| 단계 | 하는 일 | 하지 않는 일 |
|------|---------|-------------|
| **측정** | 무엇이 어떻게 되어 있는지 **사실만** 수집 | "이게 위반인가"를 말하지 않는다 |
| **판정** | 수집된 사실을 정본과 대조해 등급을 매김 | 새로 코드를 읽지 않는다 |

측정 단계에서 판정을 섞으면, 검사자가 "이건 위반 같으니 더 볼 것 없다"며 수집을 중단하거나 반대로 "애매하니 넘어가자"며 사실을 누락시킨다. 사실을 먼저 다 모으고 그 다음에 정본과 대조하면 판정 근거가 리포트에 남아 재검증이 가능하다.

**측정 실패는 위반이 아니다.** 검사 도구 자체가 못 돌았으면 `SENSOR_ERROR`로 기록하고 판정하지 않는다. 이걸 "이상 없음"으로 처리하면 검사되지 않은 곳이 통과한 것으로 둔갑한다.

| 기록 | 뜻 |
|------|-----|
| 정상 측정 → 판정 | 사실을 확인했고 등급을 매겼다 |
| `SENSOR_ERROR` | **확인하지 못했다.** 통과가 아니다. 원인을 적고 재시도 대상으로 남긴다 |
| `SKIP` | 대상이 아직 없어 검사할 것이 없다(미구현 등) |

## 4-판정 체계

불일치를 발견하면 반드시 넷 중 하나로 분류한다. "다름"에서 멈추지 않는 것이 Seam QA의 핵심이다.

| 판정 | 조건 | 증상 | 조치 |
|------|------|------|------|
| **V (위반)** | 정본이 명확한데 구현이 어김 | 갑자기 어긋남 | 구현 수정. 정본은 그대로 |
| **G (계약 공백)** | 구현들이 서로 다른데 정본이 그 사실을 안 다룸(또는 `[TODO]`) | 판정할 기준이 없음 | **정본 먼저 확정** → 통지 → 양쪽 반영. 코드를 먼저 고치지 않는다 |
| **S (정본 노후)** | 구현이 일치하는데 정본만 옛 사실 | 정상 작동하나 문서가 틀림 | 소유 에이전트가 정본 개정 + 이력 기록 |
| **D (점진 이탈)** | 개별 항목은 위반이 아닌데 **같은 방향의 이탈이 누적** | **에러 없이 품질 저하** | 누적 사례를 묶어 보고. 임계 초과 시 계약에 명시적 규칙 추가 |

G·S를 V로 뭉뚱그리면 "계약이 틀렸는데 왜 내 코드를 고치냐"는 반발이 나오고 정본 신뢰가 무너진다. **판정을 나누는 것이 거버넌스를 지킨다.**

### D(점진 이탈)를 따로 두는 이유

V·S는 특정 시점의 대조로 잡힌다. D는 안 잡힌다. 매 건은 계약 위반이 아니고 테스트도 통과하기 때문이다. **한 번의 생략이 다음 생략을 정당화하면서 구조가 서서히 무너진다.**

```
1주차  계약 경유해서 호출          정상
3주차  급해서 한 곳만 직접 호출     "한 번쯤은"     ← 개별로는 위반 아님
5주차  비슷한 자리에서 또 직접 호출  "저기도 그랬으니" ← 패턴이 됨
8주차  직접 호출이 기본이 됨        계약이 死文化
```

그래서 D는 **한 건이 아니라 추세로 판정**한다. 정본의 "점진 이탈 감시" 표에 감시 축과 임계를 미리 적어두고, QA는 매 회차 카운트를 누적한다. 임계를 넘으면 D로 올린다. 카운트가 세션을 넘어 유지되도록 QA 리포트를 남긴다.

## 심각도 — 등급에서 나온다

판정(무엇이 문제인가)과 심각도(얼마나 급한가)는 다른 축이다. 심각도는 **위반된 규칙의 등급**에서 기계적으로 나온다.

| 위반 대상 | 심각도 | 다음 행동 |
|----------|--------|----------|
| `[MUST]` 정책 (`{SEAM}-P###`) | **FAIL** | 즉시 수정. 다음 단계로 넘어가지 않는다 |
| `[SHOULD]` 지침 (`{SEAM}-G###`) | **WARN** | 수정 권고. 이유를 남기면 진행 가능 |
| 등급 없음(구현 자유) | — | 판정 대상 아님 |

**여러 건이 걸리면 가장 높은 심각도를 전체 판정으로 삼되, 위반된 Rule ID는 전부 출력한다.** 하나만 보고하면 나머지가 다음 회차에 새 문제처럼 다시 나온다.

**판정 근거에 Rule ID를 반드시 적는다.** ID가 있어야 "이 규칙에서 위반이 반복된다"는 패턴이 보이고, 그게 계약 자체를 손볼 신호가 된다.

## 절차

### 1. 검사 대상 도출 (탐색하지 않는다)
`docs/context/README.md`의 오너십 맵에서 `seam → 소유자 → 소비자 N` 을 읽는다. 검사 단위는 **(seam, 소비자) 쌍**이다. 소비자가 3명인 seam은 3건이 된다.

### 2. 계약 항목을 assertion으로 전개
정본을 읽고 **검증 가능한 단문**으로 쪼갠다. 이 전개가 Seam QA의 실질이다.

```
docs/context/api.md:
  "GET /jobs 응답은 { items: Job[], next_cursor: string|null }"
      ↓ 전개
  A1. 응답 최상위 키는 items, next_cursor 두 개다
  A2. items 원소는 Job 스키마를 만족한다
  A3. 마지막 페이지에서 next_cursor는 null이다 (부재나 "" 아님)
```
**각 assertion에 대상 Rule ID를 붙인다.** ID가 있어야 위반 시 심각도가 자동으로 정해지고 이력이 누적된다.

`[TODO]`/`[OPEN]` 항목은 assertion으로 만들지 말고 **G 후보**로 따로 적는다 — 계약이 아직 없는 곳이 곧 경계면 버그가 자라는 자리다.

### 3. 측정 — 양쪽을 동시에 열고 사실만 기록
소유자 구현과 소비자 구현을 **같은 컨텍스트에 함께 로드**한다. 따로 읽고 기억에 의존하면 shape 차이를 놓친다. "존재하는가"가 아니라 **"모양이 같은가"** 를 본다.

이 단계에서는 **위반 여부를 말하지 않는다.** assertion별로 관측된 사실만 적는다. 검사 도구가 못 돌았으면 `SENSOR_ERROR`, 대상이 아직 없으면 `SKIP`으로 기록한다 — 둘 다 "이상 없음"이 아니다.

### 4. 판정 — 사실을 정본과 대조
수집된 사실을 정본과 맞춰 V/G/S/D를 매기고, 위반된 Rule ID의 등급에서 FAIL/WARN을 정한다. 이 단계에서 코드를 새로 읽지 않는다 — 3단계에서 놓친 게 있으면 3단계로 돌아간다.

D 판정은 이번 회차만으로 나오지 않는다. 이전 리포트의 누적 카운트를 확인하고 임계 초과 여부를 본다.

### 5. 기록
`assets/seam-qa-report.template.md` 형식으로 남긴다. 판정, 심각도, Rule ID, 근거 위치(`파일:줄`), 조치 대상을 명시한다. G 판정은 `docs/context/open-decisions.md`에도 등재한다.

### 6. 재검증
수정 후 해당 (seam, 소비자) 쌍만 재실행한다. 전체 재검증은 계약이 바뀐 seam에 대해서만. `SENSOR_ERROR`가 난 항목은 원인을 해소하고 반드시 다시 측정한다 — 미측정을 남긴 채 통과 처리하지 않는다.

## 실행 타이밍 — 점진 검증

전체 완성 후 1회 실행하면 이미 오염이 퍼진 뒤다. **모듈 하나가 seam을 건드린 직후** 그 seam만 검사한다.

| 시점 | 검사 범위 |
|------|----------|
| 소유자 구현 완료 | 정본 ↔ 소유자 (계약대로 만들었나) |
| 소비자 하나 완료 | 정본 ↔ 그 소비자 |
| 계약 변경 발생 | 해당 seam의 **모든** 소비자 재검사 |
| 통합·릴리스 전 | 전체 seam 1회 스윕 |

## 경계면 버그 패턴 (seam 유형별)

정본을 assertion으로 전개할 때 아래를 우선 착안점으로 삼는다. 대부분의 경계면 버그는 "타입은 맞는데 의미가 다른" 축에서 난다.

| seam 유형 | 자주 깨지는 축 |
|-----------|---------------|
| **데이터모델** | 필드명 표기(snake/camel) · nullable 여부 · 빈값 표현(`null` vs `""` vs 키 부재) · 숫자 타입(정수/실수/문자열) · 날짜 형식과 타임존 |
| **API** | 페이지네이션 규약(cursor/offset, 종료 신호) · 에러 응답 shape(코드 체계·HTTP status 매핑) · 배열 vs 단건 래핑 · 부분 응답 시 누락 키 |
| **상태머신** | 정본에 없는 전이(무단) · 정본에 있으나 아무도 안 쓰는 전이(죽은 전이) · 상태 문자열 표기 불일치 · 종료 상태에서의 재진입 |
| **RBAC** | API 권한과 화면 권한 불일치(버튼은 숨겼는데 API는 허용) · 소유자(self) 판정 기준 · 미인증 vs 권한없음 응답 구분 |
| **지표** | 분모 정의(전체 vs 유효) · 집계 기간 경계(포함/배제) · 반올림·단위(초 vs ms, 비율 vs 퍼센트) |
| **용어(glossary)** | 같은 개념 다른 이름 · 다른 개념 같은 이름(가장 위험) |

> "버튼은 숨겼는데 API는 허용"처럼 **한쪽만 보면 정상인 결함**이 경계면 버그의 전형이다. 반드시 양쪽을 동시에 본다.

## QA 에이전트

### 타입
**`general-purpose`를 쓴다.** 읽기 전용 타입은 대조 스크립트 실행이나 리포트 작성을 못 한다. QA는 읽고 → 대조하고 → 리포트를 쓰는 역할이므로 쓰기·실행 권한이 필요하다.

### 오너십에서의 위치
QA 에이전트는 **어떤 seam도 소유하지 않는다.** 모든 정본의 소비자이며, 정본을 고칠 권한이 없다. 계약 개정이 필요하면(G·S) 소유 에이전트에게 통지할 뿐이다. 이 분리가 "QA가 계약을 자기 편한 대로 바꾸는" 사태를 막는다.

### 정의 템플릿

```markdown
---
name: seam-qa
description: "경계면 계약 대조 검증 담당. docs/context 정본과 각 소비자 구현을 동시에 읽어 사실을 모으고 V/G/S/D로 판정한다. 모듈 완성 직후·계약 변경 후·릴리스 전 검증, '검증해줘/QA/정합성/계약 위반/점검' 요청 시 사용. 재검증·부분 재실행 요청도 포함."
model: opus
---
# Seam QA — 경계면 계약 대조 검증

## 핵심 역할
docs/context 정본을 정답지로 삼아 구현과 대조하고 V/G/S/D로 판정한다. 계약을 직접 수정하지 않는다.

## 작업 원칙
- 검사 단위는 (seam, 소비자) 쌍. 오너십 맵에서 대상을 도출하고 코드 탐색에 의존하지 않는다.
- 계약 항목을 assertion 단문으로 전개하고 **각각에 Rule ID를 붙인다.**
- **측정과 판정을 분리한다.** 먼저 사실만 모으고(위반 여부를 말하지 않는다), 그 다음 정본과 대조해 판정한다.
- 소유자와 소비자 구현을 **동시에** 로드한다.
- 불일치는 반드시 V/G/S/D 중 하나로 판정한다. "다름"에서 멈추지 않는다.
- 심각도는 위반된 Rule의 등급에서 정한다 — MUST 위반 = FAIL, SHOULD 이탈 = WARN.
- 정본이 진실이다. 구현을 근거로 계약을 정답 처리하지 않는다.

## 입력/출력 프로토콜
- 입력: docs/context/{seam}.md, docs/context/README.md(오너십 맵), 소유자·소비자 구현, **직전 QA 리포트**(D 판정의 누적 카운트 확인용)
- 출력: _workspace/NN_seam-qa_{seam}.md (assets/seam-qa-report.template.md 형식)

## 팀 통신 프로토콜
- V 판정 → 구현 담당 에이전트에게 수정 요청 (FAIL이면 즉시, WARN이면 권고)
- G·S 판정 → 해당 seam 소유 에이전트에게 계약 개정 **제안** + open-decisions 등재 요청
- D 판정 → 소유 에이전트에게 누적 근거와 함께 명시적 규칙 추가 제안
- 오케스트레이터에게 판정 요약 보고 (FAIL 건수 포함)

## 측정 실패 처리
- 검사 도구가 실행되지 않음 → `SENSOR_ERROR`로 기록하고 판정하지 않는다. **통과로 처리 금지.**
- 대상이 아직 없음(미구현) → `SKIP`으로 명시 기록. 누락과 구분한다.
- 같은 항목에서 SENSOR_ERROR가 3회 연속이면 사람에게 알린다.
- 어떤 Rule과도 매칭되지 않는 관측 → 판정하지 말고 **G 후보**로 기록. 계약에 규칙 추가가 필요하다는 신호다.
```

## 팀 구성 시 배치

**생성-검증 패턴**으로 배치하되, 최종 1회가 아니라 각 seam 완성 직후 호출한다. 팀 모드에서는 QA를 상시 팀원으로 두고 `TaskCreate`의 의존 관계에 "구현 태스크 → QA 태스크"를 걸어 점진 검증을 강제한다.

## 안티패턴
- **존재 확인으로 대체** — "엔드포인트가 있다"는 검증이 아니다. shape을 본다.
- **한쪽만 읽고 판정** — 소유자만 보면 소비자 오해를 못 잡는다.
- **판정 없는 불일치 보고** — 조치 대상이 정해지지 않아 리포트가 사장된다.
- **측정하면서 판정** — 수집을 중단하거나 애매한 사실을 누락시킨다. 사실을 다 모으고 대조한다.
- **`SENSOR_ERROR`를 통과로 처리** — 검사되지 않은 곳이 정상으로 둔갑한다. 가장 위험한 오류다.
- **모든 위반을 FAIL로** — 등급 없이 다 급하면 아무것도 안 급해진다. MUST/SHOULD를 구분한다.
- **D를 한 회차로 판정** — 점진 이탈은 추세다. 누적 없이 판정하면 매번 "위반 아님"으로 지나간다.
- **QA에게 정본 수정 허용** — 계약이 QA 편의대로 흘러 SoT가 무너진다.
- **전체 완성 후 1회 실행** — 오염 전파 후의 검증은 비용이 몇 배다.
