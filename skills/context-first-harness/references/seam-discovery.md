# P-B: Seam 발견 (상세) — HE보다 먼저

**Seam = 2개 이상 소비자가 공유하는 인터페이스 사실.** 경계면 버그와 drift는 seam이 여러 곳에 다르게 존재할 때 생긴다. seam을 먼저 식별해 계약 파일 set과 오너십을 정하고, 그 소유자로 에이전트를 정의한다.

## 왜 HE보다 먼저인가
에이전트를 먼저 만들고 계약을 나중에 붙이면, 계약이 각 에이전트 산출물에 흩어져 이후 "seam-only 다이어트" 재작업이 발생한다. seam을 먼저 못박으면 에이전트가 "이 seam의 소유자"로 태어나 중복이 없다.

## Seam 판별 기준
| 질문 | seam인가? |
|------|----------|
| 2+ 에이전트/컴포넌트가 이 사실에 의존하는가? | 예 → seam(context) |
| 이 사실이 어긋나면 경계면 버그가 나는가? | 예 → seam |
| 한 소유자의 내부 구현 방법인가? | 아니오 → design |

## 도메인별 seam 카탈로그 (출발점, 고정 아님)
| 도메인 유형 | 흔한 seam |
|------------|----------|
| 웹 서비스(다중 서비스) | 용어 / 데이터모델(엔티티·제약) / API(엔드포인트·응답형식) / 상태머신 / RBAC / 지표 산식 |
| 데이터/ETL 파이프라인 | 스키마·계약 / 단계 인터페이스 / 파티션·키 규약 / 품질 지표 / 재처리 상태 |
| 콘텐츠 생성 파이프라인 | 산출물 스펙 / 단계 산출물 형식 / 톤·스타일 가이드 / 승인 상태 |
| ML 시스템 | 피처 스키마 / 라벨 규약 / 평가 지표 정의 / 모델 인터페이스 |
| CLI/라이브러리 | 공개 API 시그니처 / 설정 스키마 / 에러 코드 / 하위호환 계약 |

> 위는 출발점일 뿐. 실제 seam은 "누가 무엇을 공유하는가"로 **도출**한다. 억지로 6개를 만들지 말고, 공유되는 사실만 seam으로.

## 산출
- seam 목록(각 seam은 계약 파일 1개가 됨: `docs/context/{seam}.md`)
- **오너십 맵 초안**: seam → 소유 에이전트(1명) → 소비 에이전트(N명)
- 교차-도구 보조(계약 아님, 그래도 유지): glossary / fr-index / open-decisions

## 안티패턴
- seam에 구현 상세를 넣기(→ design). seam은 인터페이스 사실까지만.
- 소비자가 1명뿐인 것을 seam으로 승격(과분할·오버스펙). 단일 소유면 그 에이전트의 design.
- seam 없이 에이전트부터 만들기(→ 이후 재작업).
