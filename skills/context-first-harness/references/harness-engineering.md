# HE: 팀 구성 (상세) — seam 소유자로 에이전트·스킬·오케스트레이터

seam 오너십(P-B)을 바탕으로 에이전트 팀을 만든다. HE는 "누가·어떻게", CE는 "무엇". HE는 CE(정본)를 **소비/소유**하도록 배선된다(P-D).

> 이 문서는 HE의 허브다. 세부는 아래로 분기한다 — 필요한 것만 읽는다.
> - 실행 모드·아키텍처 패턴·데이터 전달·에러 핸들링: `execution-modes.md`
> - 스킬 작성(description·본문·중복 검토): `skill-authoring.md`
> - QA 에이전트 설계와 계약 대조 검증: `seam-qa.md`
> - 재실행·확장·drift 감사: `re-execution.md`

## 원칙: 에이전트는 seam의 소유자다

에이전트를 "역할"이 아니라 **"어떤 계약을 책임지는가"** 로 정의한다. seam마다 소유자가 정확히 1명이고 나머지는 소비자다. 이 구도가 잡히면 중복 에이전트가 생기지 않고, 계약 변경 시 누가 고치고 누구에게 통지할지가 자동으로 정해진다.

- 한 seam을 둘이 나눠 소유 → **금지**(SoT 붕괴). 일이 많으면 소유자 1명 + 작업자를 붙인다.
- 한 에이전트가 인접 seam 여러 개 소유 → 허용. 소유자가 1명이기만 하면 된다.
- 소유자 없는 seam → 배선 누락. P-B로 돌아간다.
- 소비자 0명인 seam → seam이 아니라 design. P-B 재검토.

## 에이전트 정의 (필수)

**모든 에이전트는 `대상/.claude/agents/{name}.md` 파일로 정의**한다(빌트인 타입이라도). 프롬프트에 역할을 인라인하지 않는다 — 파일로 존재해야 다음 세션 재사용과 협업 프로토콜이 보장된다. **모두 `model: "opus"`.**

```markdown
---
name: {agent}
description: "1-2문장 역할 + 트리거 키워드. 담당 seam 명시. 후속 작업 키워드 포함."
model: opus
---
# {Agent} — {역할}
## 핵심 역할
## 작업 원칙
## 컨텍스트 자산          (소유 정본 / 참조 정본 — 복제 금지, Read로 로드)
## 입력/출력 프로토콜      (입력 소스, 출력 경로 _workspace/…, 형식)
## 팀 통신 프로토콜        (수신/발신 대상, 계약 변경 통지 대상)
## 에러 핸들링
## 재호출 시              (이전 산출물이 있을 때의 행동 — re-execution.md)
## 협업                   (담당 seam의 소유자임을 명시)
```

**중복 검토(생성 전 필수):** `대상/.claude/agents/`를 먼저 읽는다. 역할이 겹치는 에이전트가 이름만 다르게 쌓이기 쉽다. 판단 축은 `execution-modes.md`의 "에이전트 분리 기준"(seam 오너십이 1순위).

## 스킬

에이전트마다 1~N개를 `대상/.claude/skills/{name}/SKILL.md`에 만든다. 핵심만:
- frontmatter(name, description) 필수, description은 적극적으로 + **후속 작업 키워드 포함**
- 본문 500줄 이내, 초과 시 `references/`로 분리
- **"컨텍스트 자산 참조" 섹션 필수** — 소유/참조 정본 명시
- **사실은 스킬에 적지 않는다.** 절차만 담고 사실은 정본에서 읽는다
- **커맨드(`.claude/commands/`)는 만들지 않는다**

> 작성 상세·중복 검토·안티패턴: `skill-authoring.md`

## 오케스트레이터

팀 전체를 조율하는 스킬 1개. 실행 모드·데이터 흐름·에러 핸들링·테스트 시나리오·Phase 0 컨텍스트 확인(초기/후속/부분 재실행)을 담는다.

**완료 시 규칙(중요):**
- 정본(`docs/context`)의 `[TODO]/[OPEN]`을 팀 결과로 채운다
- **design은 seam을 재기술하지 않는다** — 계약은 context를 링크만, design에는 구현 방법만
- 확정된 계약 사실이 `_workspace/`에만 남지 않게 한다(정본으로 승격)

> 패턴·템플릿·데이터 전달 프로토콜: `execution-modes.md`

## QA 에이전트 (권장)

`general-purpose` 타입. 핵심은 "존재 확인"이 아니라 **정본 ↔ 구현 대조**이며, 불일치를 V(구현 위반)/G(계약 공백)/S(정본 노후)로 **판정**한다. 어떤 seam도 소유하지 않고 계약 수정 권한이 없다. 각 모듈 완성 직후 점진 실행.

> 방법론·정의 템플릿·경계면 버그 패턴: `seam-qa.md`
