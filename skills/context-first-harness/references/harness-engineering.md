# HE: 팀 구성 (상세) — seam 소유자로 에이전트·스킬·오케스트레이터

seam 오너십(P-B)을 바탕으로 에이전트 팀을 만든다. HE는 "누가·어떻게", CE는 "무엇". HE는 CE(정본)를 **소비/소유**하도록 배선된다(P-D).

## 실행 모드
- **에이전트 팀이 기본** — 2+ 에이전트가 협업하고 실시간 조율(SendMessage)·공유 작업목록(TaskCreate)이 품질을 높일 때.
- **서브 에이전트** — 단일 작업, 결과만 반환하면 충분할 때(`Agent` 도구, `run_in_background`).
- **하이브리드** — Phase마다 다르게.

## 아키텍처 패턴 (택1 또는 복합)
| 패턴 | 언제 |
|------|------|
| 파이프라인 | 순차 의존 |
| 팬아웃/팬인 | 병렬 독립 → 통합 (팀 모드 최적) |
| 전문가 풀 | 상황별 선택 호출 |
| 생성-검증 | 생성 후 QA 검수 |
| 감독자 | 중앙이 동적 분배(공유 작업목록과 매칭) |
| 계층 위임 | 재귀 분해(2단계 이내) |

## 에이전트 정의 (필수)
**모든 에이전트는 `대상/.claude/agents/{name}.md` 파일로 정의**한다(빌트인 타입이라도). 프롬프트에 역할을 인라인하지 않는다 — 파일로 존재해야 재사용·협업 프로토콜이 보장된다. **모두 `model: "opus"`.**

```markdown
---
name: {agent}
description: "1-2문장 역할 + 트리거 키워드. 담당 seam 명시."
model: opus
---
# {Agent} — {역할}
## 핵심 역할
## 작업 원칙
## 입력/출력 프로토콜   (입력 소스, 출력 경로 _workspace/…, 형식)
## 팀 통신 프로토콜      (수신/발신 대상, 작업 요청 범위)
## 에러 핸들링
## 협업                  (담당 seam의 소유자임을 명시)
```

각 에이전트는 **자신이 소유한 seam(context 정본)의 소유자**이며, 다른 seam은 소비자로 Read한다.

## 스킬
에이전트마다 1~N개 스킬을 `대상/.claude/skills/{name}/SKILL.md`에 만든다. frontmatter(name, description) 필수, description은 적극적(pushy)으로 + 후속 작업 키워드 포함. 본문 500줄 이내, 초과 시 references/ 분리. **커맨드(.claude/commands)는 만들지 않는다.**

## 오케스트레이터
팀 전체를 조율하는 스킬 1개. 팀 구성(TeamCreate)·작업 분배(TaskCreate)·데이터 흐름·에러 핸들링·테스트 시나리오를 담는다. **완료 시 규칙(중요):**
- 정본(`docs/context`)의 `[TODO]/[OPEN]`을 팀 결과로 채운다.
- **design은 seam을 재기술하지 않는다** — 계약은 context를 링크만, design에는 구현 방법만.
- Phase 0에 컨텍스트 확인(초기/후속/부분 재실행) 단계 포함.

## QA 에이전트 (권장)
`general-purpose` 타입(스크립트 실행 가능). 핵심은 "존재 확인"이 아니라 **"경계면 교차 비교"** — 양쪽 산출물을 동시에 읽고 seam(정본)과 대조. 각 모듈 완성 직후 점진 실행(incremental QA).
