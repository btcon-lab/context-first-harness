# 후속 작업 · 재실행 (Phase 0 / Phase 9 상세)

하네스는 한 번 돌고 끝나지 않는다. 두 번째 요청부터가 실사용이며, 여기서 무너지는 하네스가 많다. 재실행이 **전체 재생성**으로 흐르면 이전 결정이 지워지고, 사용자는 하네스를 우회해 수동 작업으로 돌아간다.

## 실행 모드 판별 (오케스트레이터 Phase 0)

워크플로우 시작 시 반드시 판별한다. 판별 없이 시작하면 항상 초기 실행처럼 동작한다.

| 상태 | 판별 | 실행 |
|------|------|------|
| `_workspace/` 없음, `docs/context/` 없음 | **초기 실행** | 전체 파이프라인 |
| 산출물 있음 + 부분 수정 요청 | **부분 재실행** | 해당 에이전트만 재호출, 나머지 산출물 유지 |
| 산출물 있음 + 새 입력 제공 | **새 실행** | `_workspace/` → `_workspace_prev/`로 이동 후 전체 |
| 산출물 있음 + 계약 변경 요청 | **CE 재실행** | 아래 별도 절차 |
| 산출물 있음 + 점검 요청 | **감사** | 읽기만, 생성 없음 |

> **판단이 애매하면 사용자에게 확인한다.** 새 실행은 이전 산출물을 밀어내므로, 부분 재실행으로 충분한 요청에 새 실행을 태우면 결정이 소실된다. 되돌리기 어려운 쪽을 기본값으로 삼지 않는다.

## CE 재실행 — 계약이 바뀔 때

CFHM에서 가장 중요한 재실행 경로다. 계약 변경은 코드 재생성이 아니라 **정본 개정에서 시작**한다.

```
1. 영향 범위 산정   오너십 맵에서 그 seam의 소비자 전원을 나열
2. 개정안 작성      소유 에이전트가 초안 — 대상 Rule ID / 현재→제안 / 근거 / 영향 소비자
3. ★ 사람 승인 ★    계약은 아키텍처 결정이다. 규칙의 추가·완화·삭제는 승인 없이 반영하지 않는다
4. 정본 개정        승인된 내용만 반영 + 변경 이력 기록
5. 통지             소비자 에이전트 전원에게 SendMessage (누락 시 drift 발생)
6. 반영             각 소비자가 자기 구현을 개정된 계약에 맞춤
7. 검증             Seam QA로 해당 seam 전체 재검사 (일부가 아니라 전 소비자)
8. 기록             PROGRESS 세션 로그 + open-decisions 종결 처리
```
> 승인 없이 진행 가능한 예외: `[TODO]`를 처음 채우는 경우, 오타·서식·링크 교정. 상세는 `governance.md`.

**금지:** 구현을 먼저 고치고 정본을 나중에 맞추기. 이 순서가 한 번 허용되면 정본은 "뒤따라가는 문서"가 되어 신뢰를 잃는다.

**부분 재실행과의 차이:** 부분 재실행은 한 에이전트만 다시 돌리면 되지만, CE 재실행은 **소비자 전원이 대상**이다. 계약은 공유 사실이므로 한 명만 반영하면 그 자체가 경계면 버그다.

## 산출물 보존 규칙

| 대상 | 재실행 시 |

| `docs/context/` (정본) | **덮어쓰지 않는다.** 개정만. 확정된 계약을 재생성으로 날리면 결정 이력이 사라진다 |

## 확장 시 Phase 선택

기존 하네스에 무언가를 더할 때 전체를 다시 돌리지 않는다.

| 변경 유형 | P-A | P-B | HE | P-C | P-D | P-E | P-F | 검증 |
|----------|-----|-----|----|----|-----|-----|-----|------|
| 에이전트 추가 | – | 담당 seam 확인 | 필수(중복 검토) | 신규 seam 있으면 | 필수 | 오너십 맵 갱신 | 이력 | 구조+트리거 |
| 스킬 추가·수정 | – | – | 필수(중복 검토) | – | 참조 변경 시 | – | 이력 | 구조+트리거 |
| **seam 추가** | – | **필수** | 소유자 지정 | **필수** | 필수 | 필수 | 이력 | 전체 |
| 계약 변경 | – | – | – | 개정 | – | 이력 | 이력 | **Seam QA 전체** |
| 원문 개정 | 재생성 | 재검토 | – | 영향분 개정 | – | – | 이력 | Seam QA |
| 아키텍처 변경 | – | 필수 | 필수 | 영향분 | 필수 | 필수 | 이력 | 전체 |

## 에이전트 정의에 넣을 재호출 지침

각 에이전트 `.md`에 "이전 산출물이 있을 때의 행동"을 명시한다. 없으면 매번 백지에서 시작한다.

```markdown
## 재호출 시
- `_workspace/`에 내 이전 산출물이 있으면 **읽고 개선**한다. 처음부터 다시 만들지 않는다.
- 사용자 피드백이 지정되면 **해당 부분만** 수정하고 나머지는 보존한다.
- 내가 소유한 정본은 개정만 한다. 재생성하지 않는다.
- 참조 정본이 바뀌었으면(변경 이력 확인) 내 산출물에 반영한다.
```

## 오케스트레이터 description에 넣을 후속 키워드

초기 생성 키워드만 있으면 두 번째 요청부터 트리거되지 않는다. 반드시 포함한다:
"다시 실행", "재실행", "업데이트", "수정", "보완", "이어서", "이전 결과 기반으로", "{도메인}의 {부분}만 다시", "계약 바꿔줘", "점검"

## Drift 감사 (운영·유지보수)

Phase 0에서 "운영" 분기로 들어왔을 때 수행한다.

1. `.claude/agents/` 실제 목록 ↔ 오케스트레이터가 아는 구성 → 불일치 목록
2. `.claude/skills/` 실제 목록 ↔ 오케스트레이터 구성 → 불일치 목록
3. **오너십 맵의 seam ↔ `docs/context/` 실제 파일** → 고아 정본(소유자 없음), 유령 seam(파일 없음)
4. **정본 ↔ 구현** → Seam QA 스윕
5. CLAUDE.md 변경 이력 ↔ 실제 파일 상태
6. 감사 결과를 사용자에게 보고하고 조치 계획을 확인받는다

변경은 **한 번에 하나씩**, 각 변경 후 즉시 이력을 갱신하고 검증한다. 몰아서 고치면 무엇이 무엇을 깨뜨렸는지 추적 불가능해진다.

## 진화 트리거

사용자가 "고쳐줘"라고 할 때만이 아니라, 다음이 관찰되면 하네스 개선을 먼저 제안한다.
- 같은 유형의 피드백이 2회 이상 반복
- 특정 에이전트가 반복적으로 같은 실패
- 사용자가 오케스트레이터를 우회해 수동 작업
- **Seam QA에서 같은 seam의 G 판정이 반복** → 계약이 실제 경계를 못 담고 있다. P-B 재검토 신호
