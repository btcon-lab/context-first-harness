# Context-First Harness (CFHM)

> Claude Code 플러그인. **"이 계획서로 하네스 만들어줘"** 한마디로 에이전트 팀을 꾸려주는데, 팀보다 **약속(계약) 문서를 먼저** 만듭니다.

## 이런 일, 겪어보셨나요

백엔드는 `user_id`로 내려주고, 프론트는 `userId`로 받습니다. 각자 코드만 보면 둘 다 멀쩡합니다. 테스트도 통과합니다. 붙이는 순간 터집니다.

에이전트를 여러 개 돌릴 때도 똑같습니다. 각자 자기가 아는 대로 만들고, 합칠 때가 되어서야 서로 다르게 알고 있었다는 걸 압니다.

**그래서 CFHM은 순서를 바꿉니다.** 에이전트를 만들기 전에 "이건 이렇게 하기로 한다"는 약속을 문서로 먼저 못박고, 그 약속의 **주인**으로 에이전트를 세웁니다. 나중에 서로 어긋나면 코드가 아니라 약속 문서가 정답입니다.

## 먼저 알아둘 말 네 가지

이 문서와 플러그인 안에서 계속 나오는 말입니다. 어려운 개념은 아닙니다.

| 말 | 뜻 |
|---|---|
| **경계면** (seam) | 두 군데 이상이 같이 쓰는 약속. API 응답 모양, 상태값 이름, 권한 규칙처럼 한쪽만 바꾸면 다른 쪽이 깨지는 것들입니다. 한 곳에서만 쓰는 건 경계면이 아닙니다. |
| **정본** | 그 약속을 적어둔 **기준 문서**. `docs/context/` 아래에 경계면마다 하나씩 둡니다. 코드와 문서가 다르면 이 문서가 맞는 걸로 칩니다. |
| **주인** (소유자) | 경계면 하나마다 책임지는 에이전트 한 명. 그 사람만 정본을 고치고, 고치면 쓰는 쪽에 알립니다. |
| **어긋남** (drift) | 같은 내용이 여러 군데 적혀 있다가 한쪽만 바뀌어서 안 맞게 된 상태. CFHM이 막으려는 게 이겁니다. |

## 무엇을 해주나요

계획서 한 부(또는 한 문장)를 주면 순서대로 이렇게 합니다.

1. **긴 문서는 잘라둡니다** — 매번 통째로 읽으면 토큰이 아까우니, 한 번만 쪼개서 각자 필요한 부분만 보게 합니다.
2. **경계면을 찾습니다** — "여기서 두 군데 이상이 같이 쓰는 게 뭐지?"를 뽑아내고, 각각 누가 책임질지 정합니다. ★ **이걸 팀 짜기보다 먼저 합니다**
3. **팀을 만듭니다** — 에이전트와 스킬을 만듭니다. 각 에이전트는 자기가 맡은 경계면의 주인입니다.
4. **약속 문서를 채웁니다** — 경계면마다 기준 문서를 하나씩 만들고, 아직 안 정해진 건 `[TODO]`로 남깁니다.
5. **서로 연결합니다** — 각 스킬이 어떤 기준 문서를 보고 일할지 적어둡니다.
6. **운영 규칙을 적습니다** — 약속을 바꿀 때 밟을 순서(문서 먼저 → **사람 승인** → 알리기 → 반영 → 확인 → 기록)를 정해둡니다.
7. **작업 일지를 만듭니다** — `PROGRESS.md`에 지금 어디까지 왔는지 남기고, 원하면 갱신을 깜빡하지 않게 막아주는 훅도 깔아줍니다.
8. **감시를 걸어둡니다(선택)** — 코드가 바뀌면 관련 검사가 자동으로 돌게 합니다.

마지막으로 제대로 만들어졌는지 확인하고, 쓰면서 고쳐나갑니다.

## 왜 팀보다 약속이 먼저인가

에이전트부터 만들면 약속이 각자 결과물 여기저기에 흩어집니다. 나중에 "이건 공통 약속이니 한 곳으로 모으자"며 갈아엎게 됩니다. 반대로 약속을 먼저 못박으면 에이전트가 처음부터 "이 약속의 주인"으로 태어나서 겹치는 일이 없습니다.

## 쓰기 전에 확인할 것

| 항목 | 필요한가 | 없으면 |
|---|---|---|
| **에이전트 팀 기능** | 팀 모드를 쓰려면 필요 | 환경변수 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`이 켜져 있어야 팀 도구(`TeamCreate` 등)를 씁니다. 꺼져 있으면 **서브 에이전트 모드로 대신 진행**되며, 나머지 기능은 그대로 동작합니다. ([공식 문서](https://code.claude.com/docs/en/agent-teams)) |
| **python3** | 계획서를 쪼갤 때만 | 표준 라이브러리만 씁니다. 별도 설치 필요 없습니다. |
| **bash** | 훅을 깔 때만 | 훅은 전부 선택 기능입니다. |
| **jq** | 자동 감시를 쓸 때만 | 작업 일지 훅은 `jq` 없이도 돕니다. 다만 **자동 감시(선택 기능)는 `jq`가 필요**하고, 없으면 그 훅만 조용히 건너뜁니다. |

## 설치

### 1) 마켓플레이스로 설치 (가장 간단)

```
/plugin marketplace add btcon-lab/context-first-harness
/plugin install context-first-harness@cfh-marketplace
```

### 2) 직접 clone해서 쓰기 (포크·수정해서 쓸 때)

```shell
git clone https://github.com/btcon-lab/context-first-harness.git
cd context-first-harness
```

clone한 폴더를 그대로 마켓플레이스로 등록하면 됩니다.

```
/plugin marketplace add ./context-first-harness
/plugin install context-first-harness@cfh-marketplace
```

플러그인 없이 스킬만 쓰고 싶다면 복사해도 됩니다.

```shell
# 모든 프로젝트에서 쓰기
cp -r skills/context-first-harness ~/.claude/skills/context-first-harness

# 특정 프로젝트에서만 쓰기
cp -r skills/context-first-harness /경로/내프로젝트/.claude/skills/context-first-harness
```

> **설치했다고 바로 쓸 수 있는 게 아닙니다.** 스킬은 세션이 시작될 때 읽어들이기 때문에, 방금 설치한 그 세션에서는 잡히지 않습니다. Claude Code를 껐다 켜거나 `/clear`로 새 세션을 여세요.
>
> `/reload-plugins`는 플러그인 목록만 다시 읽습니다. 이미 열려 있는 세션의 스킬 목록에는 반영이 안 될 수 있으니, 설치 직후에는 새 세션을 여는 쪽이 확실합니다.

## 써보기

새 세션에서 이렇게 말하면 됩니다. 아래 문장을 그대로 복사해 쓰세요.

**계획서가 있을 때**
```
이 개발 계획서로 하네스를 구성해줘. (파일 첨부 또는 경로)

```

**계획서 없이 한 문장으로**
```
크롤링 서비스 하네스를 만들어줘. 수집기, API 서버, 관리 화면이 있고

```

**이미 만든 하네스를 점검할 때**
```
하네스 점검해줘. 기준 문서랑 실제 코드가 어긋난 데 있는지 봐줘.
```

**약속을 바꿔야 할 때**
```
작업 상태에 '보류'를 추가해야 해. 기준 문서부터 고치고
```


## 경계면 버그를 잡는 법

만든 다음이 더 중요합니다. 약속과 실제 코드가 어긋나는 걸 잡는 게 Seam QA입니다.

보통의 검증은 **코드끼리** 비교합니다. 그런데 이러면 "둘이 다르다"까지만 알 수 있고, **어느 쪽이 틀렸는지는 알 수 없습니다.** 결국 사람에게 "어느 게 맞아요?"라고 되묻게 됩니다. 게다가 어디가 경계면인지부터 코드를 뒤져 찾아야 하고, 못 찾은 곳은 영영 검사되지 않습니다.

CFHM은 **기준 문서와 코드**를 비교합니다. 기준 문서가 정답이니 판정이 나옵니다. 검사할 곳도 "누가 뭘 책임지는지" 표에 이미 적혀 있어서 찾아다닐 필요가 없습니다.

어긋난 걸 찾으면 넷 중 하나로 **판정**합니다.

| 판정 | 어떤 경우 | 어떻게 |
|---|---|---|
| **V** 코드가 어겼다 | 문서는 분명한데 코드가 다르게 만듦 | 코드를 고칩니다 |
| **G** 약속이 없었다 | 문서에 그 얘기가 아예 없거나 `[TODO]` | 문서부터 정하고, 그 다음 코드를 맞춥니다 |
| **S** 문서가 낡았다 | 코드들은 다 맞는데 문서만 옛날 내용 | 주인이 문서를 고칩니다 |
| **D** 조금씩 벗어났다 | 한 건씩 보면 위반이 아닌데 같은 방향으로 쌓임 | 누적을 근거로 약속에 규칙을 새로 넣습니다 |

넷을 뭉뚱그려 "코드 고쳐라"로 처리하면, 개발자는 "문서가 틀렸는데 왜 내 코드를 고치냐"고 합니다. 그렇게 한 번 밀리면 기준 문서를 아무도 안 믿습니다. **판정을 나누는 게 규칙을 지키는 방법입니다.**

**D가 제일 까다롭습니다.** 에러도 안 나고 테스트도 통과하기 때문입니다.

```
1주차  약속대로 호출              정상
3주차  급해서 한 곳만 질러감       "한 번쯤은"
5주차  옆자리에서도 질러감         "저기도 그랬으니"
8주차  지르는 게 기본이 됨         약속이 죽음
```

한 번씩 보면 아무 문제가 없어서, 매 시점의 검사로는 절대 안 잡힙니다. 그래서 D는 **한 건이 아니라 추세로** 봅니다. 무엇을 감시할지 미리 정해두고 횟수를 세다가 정해둔 선을 넘으면 올립니다.

### 얼마나 급한지는 등급으로

약속의 모든 문장이 똑같이 중요하진 않습니다. 그래서 항목마다 등급을 붙입니다.

| 등급 | 뜻 | 어겼을 때 |
|---|---|---|
| **MUST** | 반드시 지켜야 함. 예외 없음 | **FAIL** — 바로 고칩니다 |
| **SHOULD** | 지키는 게 좋음. 이유를 남기면 예외 가능 | **WARN** — 권고만 합니다 |
| 등급 없음 | 알아서 하면 됨 | 검사 대상이 아닙니다 |

등급이 없으면 전부 다 급해지고, 전부 급하면 결국 아무것도 안 급해집니다.

### 못 본 것과 이상 없는 것은 다릅니다

검사 도구가 안 돌아갔으면 `SENSOR_ERROR`로 남기고 **통과로 처리하지 않습니다.** 이걸 "이상 없음"으로 넘기면 아무도 안 본 곳이 검사를 통과한 것처럼 되어버립니다. 가장 위험한 실수라서 따로 표시합니다.

## 약속을 바꿀 때는 사람이 승인합니다

검사에서 문제가 나오면 할 일이 두 갈래입니다.

| | 코드를 고칠 때 | 약속을 고칠 때 |
|---|---|---|
| 언제 | **V** — 약속은 분명한데 코드가 어김 | **G · S · D** |
| 승인 | 필요 없습니다. 약속이 이미 정답이니까요 | **사람이 봐야 합니다** |

약속을 바꾸는 건 설계 결정입니다. AI가 검사 결과만 보고 기준 문서를 고치게 두면, **"코드에 맞춰서 약속을 낮추는" 쪽으로 흘러갑니다.** 몇 번만 반복돼도 기준 문서는 코드를 나중에 요약한 것에 불과해지고, 애초에 문서를 둔 이유가 사라집니다.

그래서 이렇게 갑니다.

```
문제 발견 → AI가 [제안] → 담당자가 초안 → ★사람 승인★ → 반영 → 쓰는 쪽에 알림
```

오타 고치기나 빈칸(`[TODO]`) 처음 채우기는 승인 없이 해도 됩니다. 하지만 **규칙을 느슨하게 만들거나 없애는 건 가장 엄격하게 봅니다.** 같은 규칙이 자꾸 안 지켜진다는 건 규칙이 잘못됐다는 뜻일 수도 있고, 팀이 못 지키고 있다는 뜻일 수도 있습니다. 둘은 정반대 처방이 필요한데, 그 판단은 사람이 해야 합니다.

## 코드가 바뀌면 자동으로 재봅니다 (선택)

통합할 때가 되어서야 어긋난 걸 알면 이미 늦습니다. 그래서 **파일이 바뀌는 즉시 관련 검사만 골라서** 돌리는 감시를 걸 수 있습니다.

```
코드 수정 → 관련 검사 실행 → 사실 기록 → 약속과 대조해 판정 → 고치기
```

`.claude/seam-sensors.json`에 "이 경로가 바뀌면 이 검사를 돌려라"를 적어두면 됩니다. 전부 매번 돌리면 느려서 아무도 안 쓰니까, **바뀐 것과 관련된 것만** 돕니다.

여기서도 원칙은 같습니다 — **검사는 재기만 하고, 위반인지는 판정하지 않습니다.** 검사가 실패했다고 곧 위반은 아닙니다(테스트 환경 문제일 수도 있으니까요). 사실을 모아두면 그걸 약속 문서와 대조해서 판정합니다.

seam이 몇 개 안 되는 작은 프로젝트에는 과합니다. 안 쓰면 그만이고, 설정 파일이 없으면 아무 일도 일어나지 않습니다.

## 실패했을 때는 조건을 바꿔서 다시 시킵니다

똑같은 지시로 다시 시키면 똑같이 실패합니다. 그래서 재시도할 때마다 정보를 한 단계씩 더 줍니다.

| 회차 | 더 주는 것 |
|---|---|
| 1차 | 뭐가 왜 틀렸는지 |
| 2차 | + 봐야 할 문서를 콕 집어서 |
| 3차 | + 이미 잘 된 다른 사례 |
| 그 다음 | 사람에게 넘깁니다 |

3번까지 하고 안 되면 정보가 부족한 게 아니라, 일이 너무 크거나 약속 자체가 애매한 겁니다. 무한정 반복하는 건 시간만 씁니다.

## 언제 쓰고, 언제 안 쓰나

**쓸 만합니다** — 서비스가 여러 개로 나뉘어 있고 서로 주고받는 게 많을 때. 몇 주 이상 이어지는 작업이라 지난주에 뭘 정했는지 기억이 안 날 때. 사람이 여럿이라 각자 다르게 알고 있을 위험이 클 때.

**과합니다** — 에이전트 하나면 끝나는 일. 혼자 하는 짧은 작업. 이럴 땐 일반 하네스나 스킬 하나로 충분합니다.

```
.claude-plugin/{marketplace.json, plugin.json}
skills/context-first-harness/
├── SKILL.md              메타 스킬 (전체 흐름)
├── references/           단계별 상세 안내 12종
│   ├── 약속  phase-input-curation · seam-discovery · governance
│   ├── 팀    harness-engineering(허브) · execution-modes · skill-authoring
│   ├── 검증  verification · skill-testing · seam-qa
│   └── 운영  progress-and-hooks · re-execution · control-loop
├── assets/               문서 서식 9종 (기준 문서 · 용어집 · 작업 일지 ·
│                         QA 리포트 · 센서 설정 · 훅 설정 등)
└── scripts/              split-source.py · seam-sensor-run.sh ·
                          session-start-stamp.sh · progress-stop-check.sh
```

> `references/`와 `assets/` 안에서는 위 표의 용어(경계면=seam, 기준 문서=정본, 연결=배선)를 짧은 형태로 씁니다. Claude가 읽는 문서라 간결한 쪽을 택했습니다.
